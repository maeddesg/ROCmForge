## Overview

Add professional benchmark infrastructure to ROCmForge with three components:
1. Criterion-based kernel benchmarks with statistical rigor
2. Real model benchmark with end-to-end and kernel-level profiling
3. Automated performance comparison reports

This completes the original plans' "could be improved" items and provides
regression detection, professional documentation, and comprehensive performance
analysis.

## Goals

- **Regression detection:** Catch performance degradation during development
- **Professional reports:** Publication-ready performance comparisons
- **Comprehensive coverage:** Kernel-level + end-to-end + quantization type comparison
- **No CI/CD:** Manual execution, but with automated report generation

## Architecture

### Component 1: Criterion Kernel Benchmarks

**Location:** `benches/kernels.rs` (NEW, separate from existing `cpu_gemv.rs`)

**Purpose:** Statistical benchmarking of core kernels with regression detection

**Note:** Existing `benches/cpu_gemv.rs` uses a simple manual timing approach.
The new `kernels.rs` uses Criterion for statistical rigor. Both files serve
different purposes and will coexist.

**Benchmarks:**
| Benchmark | Description | Comparison |
|-----------|-------------|------------|
| `gemv_q4k_q8_avx2` | Q4_K × Q8_K GEMV (AVX2) | vs scalar |
| `gemv_q4k_q8_scalar` | Q4_K × Q8_K GEMV (scalar) | baseline |
| `gemm_q4k_q8_avx2` | Q4_K × Q8_K GEMM (AVX2) | vs scalar |
| `gemv_q5_0_q8_0` | Q5_0 × Q8_0 operations | quantization comparison |
| `dequant_q4k` | Q4_K dequantization | per-type comparison |
| `dequant_q5_0` | Q5_0 dequantization | per-type comparison |

**Note on Q5_0:** Q5_0 kernels are already implemented (`src/cpu/kernels/gemm_q5_0_q8.rs`)
and tested (`test_q5_0_dequant`, `gemm_q5_0_transposed_produces_correct_output`).

**Output:**
- Console: Live progress + summary table
- Files: `target/criterion/` with HTML reports (JSON available via `--save-baseline`)

**Usage:**
```bash
cargo bench --bench kernels
# View: target/criterion/report/index.html
```

### Component 2: Real Model Benchmark

**Location:** `examples/benchmark_real_model.rs`

**Purpose:** End-to-end inference performance on real GGUF models with profiling

**CLI Arguments (using `clap`):**
```
benchmark_real_model [OPTIONS]

OPTIONS:
    --model-dir <DIR>         Model directory [default: $ROCMFORGE_MODEL_DIR or ./models]
    --model <PATTERN>          Filter models by glob pattern [default: *.gguf]
    --iterations <N>           Number of inference runs [default: 3]
    --tokens <N>               Tokens to generate [default: 10]
    --profile                  Enable per-layer profiling
    --output <FILE>            Output markdown file [default: docs/benchmarks/real-model-YYYY-MM-DD.md]
    -h, --help                 Print help
```

**Model Discovery:**
1. Check `--model-dir` argument
2. Check `ROCMFORGE_MODEL_DIR` environment variable
3. Fall back to `/home/feanor/Projects/Memoria/models/`
4. Fall back to `./models/` in current directory

**Available models in default location:**
- `llama3-8b-q4_k_m.gguf` - Q4_K quantized (main focus)
- `qwen2.5-0.5b-q4_k_m.gguf` - Q4_K quantized
- `qwen2.5-0.5b-instruct-q4_0.gguf` - Q4_0 quantized
- `ggml-model-Q8_0.gguf` - Q8_0 quantized

**Measurements:**
- **End-to-end:** Total time, tokens/sec, prefill ms, decode ms
- **Per-layer:** Forward pass timing for each transformer layer (with `--profile`)
- **Memory:** Peak memory usage via `sysinfo` crate (optional, Linux-only)
- **Kernel:** Detected via `CpuFeatures::get().kernel` (logged at start)

**Kernel detection:**
```rust
let features = CpuFeatures::get();
eprintln!("Kernel: {:?}", features.kernel);
// Logs: Avx, Avx2, Scalar, etc.
```

**Memory measurement:**
```rust
use sysinfo::{System, SystemExt};
let mut sys = System::new_all();
sys.refresh_memory();
let peak = sys.used_memory();
// Note: This is process memory, not peak allocation tracking
```

**Output:**
- Console: Real-time progress + summary table
- File: `docs/benchmarks/real-model-YYYY-MM-DD.md`

### Component 3: Performance Comparison Report

**Location:** `src/bench/reporter.rs`, `examples/generate_report.rs`

**Purpose:** Aggregate benchmark results into publication-ready report

**Data Sources:**
1. **Criterion data:** `target/criterion/*/benchmark.json` (generated by Criterion)
2. **Real model data:** `docs/benchmarks/real-model-*.md` (parsed for key metrics)

**Input formats:**
- Criterion: JSON files in `target/criterion/<name>/new/` directory
- Real model: Markdown files with key-value pairs in code blocks

**Report Sections:**
1. **Executive Summary** - Key findings, recommendations
2. **Kernel Performance** - AVX2 vs Scalar speedup tables
3. **Quantization Comparison** - Q4_K vs Q5_0 vs Q6_K vs Q8_0
4. **Real Model Results** - End-to-end latency by model
5. **Recommendations** - Which quantization for each use case

**Output:**
- Markdown: `docs/benchmarks/PERFORMANCE_REPORT_YYYY-MM-DD.md`
- CSV: `docs/benchmarks/data.csv` (raw data)

**CSV Schema:**
```csv
timestamp,benchmark_name,kernel_type,quantization,operation,dimension,throughput_ms,latency_ns,speedup_vs_baseline,git_commit
2026-03-27T10:00:00Z,gemv_q4k_q8,AVX2,Q4_K,gemv,896x896,0.045,45000,2.91,a2f6a77
2026-03-27T10:00:00Z,gemv_q4k_q8,Scalar,Q4_K,gemv,896x896,0.131,131000,1.00,a2f6a77
```

**CLI Arguments:**
```
generate_report [OPTIONS]

OPTIONS:
    --criterion-dir <DIR>      Criterion data directory [default: target/criterion]
    --real-model-dir <DIR>     Real model results directory [default: docs/benchmarks]
    --output <FILE>            Output report file [default: docs/benchmarks/PERFORMANCE_REPORT_YYYY-MM-DD.md]
    --include-graphs           Include ASCII graphs in report
    -h, --help                 Print help
```

## Error Handling

All errors:
- Go to stderr with clear actionable messages
- Include context (which file, which operation)
- Suggest fixes when possible

**Specific cases:**
- No models found → List searched paths, supported formats, check `--model-dir`
- Invalid dimensions → Skip with warning, log dimensions
- Feature detection failed → Log detected features, fall back to scalar
- Out of memory → Suggest reducing `--iterations` or using smaller model
- Missing Criterion data → Run `cargo bench --bench kernels` first
- Invalid model directory → Check path exists and contains `.gguf` files

## Testing

**Unit tests** (`src/bench/`):
- Model discovery finds expected models in default location
- Report generation produces valid markdown
- Timing utilities are accurate (compare to known benchmarks)
- CSV schema matches specification

**Integration test:**
- Run kernel benchmarks: `cargo bench --bench kernels`
- Run real model benchmark: `cargo run --release --example benchmark_real_model`
- Run report generation: `cargo run --release --example generate_report`
- Verify all outputs are created and valid

**Validation criteria:**
- Criterion: All benchmarks finish, produce JSON/HTML, confidence intervals shown
- Real model: Benchmarks ≥2 different model files (Q4_K and Q8_0)
- Report: Creates both markdown and CSV with valid formatting
- No regression: AVX2 shows speedup ≥1.5x over scalar baseline
- Memory: Reports peak memory (Linux-only, optional on other platforms)

## Usage Examples

**Run kernel benchmarks:**
```bash
cargo bench --bench kernels
# View HTML: target/criterion/report/index.html
# Save baseline: cargo bench --bench kernels -- --save-baseline main
```

**Benchmark real models:**
```bash
# Default (all models in default location)
cargo run --release --example benchmark_real_model

# Custom location and pattern
cargo run --release --example benchmark_real_model -- --model-dir ./models --model "*q4_k*"

# With profiling
cargo run --release --example benchmark_real_model -- --profile --iterations 5
```

**Generate full report:**
```bash
# Default (reads from target/criterion and docs/benchmarks)
cargo run --release --example generate_report

# Custom output location
cargo run --release --example generate_report -- --output /tmp/report.md
```

## Success Criteria

1. Criterion benchmarks produce statistical reports with confidence intervals
2. Real model benchmark successfully benchmarks ≥2 models (Q4_K + Q8_0)
3. Performance report generates valid markdown with comparison tables
4. CSV output matches specified schema
5. All benchmarks complete without errors on AMD Ryzen (test system)
6. AVX2 kernels show measurable speedup over scalar baseline (≥1.5x)

## Implementation Notes

- No changes to existing kernel code (`src/cpu/kernels/`)
- Existing `benches/cpu_gemv.rs` remains unchanged (simple manual benchmark)
- Criterion runs in dev mode (fast) and release mode (accurate)
- Real model benchmark requires release build for meaningful timing
- Reports are version-controlled (commit with date stamp)
- Memory measurement uses `sysinfo` crate (Linux: process memory, macOS: similar, Windows: basic)
- Can re-run and regenerate reports incrementally
